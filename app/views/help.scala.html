Introduction

When there's a significant issue or incident impacting developers, all staff, external users or impacts of other scopes, it's useful to have a communication mechanism that:
  1. what the scope is
  2. indicates who is primarily responsible for resolving the issue
  3. which team is focused on it with the primary
  4. indicates when to expect an update
  5. enables designating groups of users ("teams") to be subscribed the issue to receive communications about it
  6. clearly indicates the duration of the incident

Creating Incidents

To create an incident, click on the [New Incident] button on the main page.
  1. give the incident a Title, this will appear in the subject line of email messages concerning the incident
  2. indicate the scope, either Internal or External
  3. provide some narrative in the Description about the incident
    if it's impact is External indicate which users are impacted (e.g. just perks users, users in the U.S., all users)
  4. indicate when the Next Update should be expected
  5. The Issue Type is a mechanism to optionally link external tracking systems
    • JIRA
    • Pager Duty
    ...or it may be unrelated to an externally tracked issue and is simply a Directly reported one.
  6. assign a Primary Responder (yourself, if applicable)
  7. indicate a Response Team, if applicable
  8. determine which Teams should be Subscribed to the issue
  9. ... and Submit it!

Internal issues are typically of a "Heads Up" informational nature concerning infrastructure maintenance, significant application deployments or other system changes that will likely only be apparent to those inside the organization. The subject line in the mail messages from internal impacting issues will be prefixed "HEADSUP".

External issues are those that impact the service in a way that users will likely notice such as data publishes missing their schedule, degraded service performance, network outages, outbound mail system malfunctions. The subject line from external impacting incidents will be prefixed "SITEIMPACT".

Updating An Issue

Incidents may have updates posted to them to provide progress updates, new information and resolution details. Updates are visible in reverse chronological order on the incident's detail page. You can add one by clicking [Add Update] an incident's detail page.

Users

This app is intentionally built with nothentication - "logging in" is just saying who you are, not proving it. So this assumes a level of trust that users of the system won't misuse it. If that becomes a problem, some authentication system will be plugged in but in the meantime, we'll assume that we're all adults who find no thrill in abusing communication mechanisms.

Deployment to a trusted network is assumed; do not run this app with open public access because it can and will be abused.

Gravatars

If a user has a Gravatar account with an email address that matches their email address in this system, their avatar will be visible. Otherwise, a placeholder avatar will be shown instead.

Teams

Teams arbitrary groupings of users that can be subscribed to an issue. The responding team is automatically subscribed but others may be subscribed if the incident potentially impacts them. Teams can have many users, users can be on many teams. Incidents can have many teams subscribed, ergo many users can be kept up to date on an issue with a good deal of flexibility. A user on multiple teams subscribed to the same incident will only be sent each update only once; the superset is addressed, not a joined lists with duplicates.

Longer Term Goals

  1. Nag primary responders who don't update within the time indicated by the Next Update time
  2. Measure incident lifecycle times with correlated systems (JIRA, Pager Duty, etc)
  3. Provide a searchable knowledge respository so that resolutions to new incidents can be hastened by old ones
  4. Aid in retrospectives and post-mortems by reconstructing timelines and helping turn incidents into learning opportunities
